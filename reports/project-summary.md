### 3.5 Task 012 经验

**问题**：
Task 012 重构时，Developer 无意中将 `taskInput.value` 改为 `taskInput.value.trim()`，导致用户输入的首尾空格被丢弃，改变了历史功能行为。

**发现**：
Tester 通过独立验收发现了这个问题，证明了独立验收的价值。

**修复**：
Developer 恢复了原设计，`trim()` 只用于空输入校验判断，不改变最终保存的用户输入内容。

**经验**：
重构任务必须做历史功能回归检查；对"用户输入"保持原值，不要擅自使用 trim()、toLowerCase() 等会改变原始值的方法。

## 7. 当前多 Agent 协作流程

### 7.1 标准工作流程

```
需求文档
   │
   ▼
Planner Agent ──► 任务清单 + 验收标准
   │
   ▼
主 Agent ──► 选择下一个任务
   │
   ▼
Developer Agent ──► 开发 + 开发报告
   │
   ▼
Tester Agent ──► 独立验收 + 测试报告
   │
   ▼
测试通过?
   │
   ├─ 是 ──► 更新状态 ──► 下一个任务
   │
   └─ 否 ──► 修正循环（最多 3 轮）
```

### 7.2 核心规则

**Developer 完成不等于任务完成**：
- Developer 完成 → 生成开发报告 → 等待 Tester 验收
- Tester 验收 → 生成测试报告 → 判定 PASS 或 FAIL
- **只有 Tester PASS 后，任务才算真正完成**

**修正循环规则**：
- 如果 Tester FAIL → 进入修正循环
- 最多 3 轮修正
- 谁写的 bug 谁修复（resume 原 Developer）
- 谁提出的 bug 谁复测（resume 原 Tester）

**信息交接规则**：
- 子 Agent 有独立上下文，不能依赖彼此记忆
- 项目文件系统是 Agent 之间的共享工作区
- Agent 之间通过 docs/、reports/、memory/ 交接信息

## 8. 当前 Agent 角色体系

### 8.1 主 Agent (Coordinator)

**职责**: 任务调度、流程控制、决策协调

**关键原则**:
- 不直接编码
- 不直接测试
- 只做调度和决策

### 8.2 计划 Agent (Planner)

**职责**: 任务拆分、依赖分析、验收标准规划

**关键原则**:
- 不编码
- 不测试
- 不创建业务代码文件

### 8.3 开发 Agent (Developer)

**职责**: 独立完成分配的开发任务

**关键原则**:
- 只处理一个明确任务
- 最小必要修改
- 不擅自扩展功能
- 不修改任务状态

### 8.4 测试 Agent (Tester)

**职责**: 独立验收开发成果

**关键原则**:
- 不能只看开发报告就 PASS
- 必须读取实际代码并测试
- 不修改业务代码
