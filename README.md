## 4. 多 Agent 协作角色

### 4.1 主 Agent (Coordinator)

**职责**: 任务调度、流程控制、决策协调

**核心任务**:
- 解析任务列表，选择下一个可执行任务
- 将任务分配给开发 Agent
- 协调测试 Agent 进行验收
- 根据测试结果决定下一步行动
- 更新任务状态和工作日志

**关键原则**:
- 不直接编码
- 不直接测试
- 只做调度和决策

### 4.2 计划 Agent (Planner)

**职责**: 任务拆分、依赖分析、验收标准规划

**核心任务**:
- 将需求拆分为可独立完成的小任务
- 定义任务依赖关系
- 制定清晰、可测试的验收标准
- 评估任务优先级

**关键原则**:
- 不编码
- 不测试
- 不创建业务代码文件

### 4.3 开发 Agent (Developer)

**职责**: 独立完成分配的开发任务

**核心任务**:
- 读取任务文档和验收标准
- 进行最小必要修改
- 进行自测（基本功能验证）
- 编写开发报告

**关键原则**:
- 只处理一个明确任务
- 不擅自扩展功能
- 不修改任务状态

### 4.4 测试 Agent (Tester)

**职责**: 独立验收开发成果

**核心任务**:
- 根据验收标准逐项测试
- 读取实际代码，不盲信开发报告
- 记录测试结果
- 发现问题时提供详细反馈

**关键原则**:
- 不能只看开发报告就 PASS
- 必须读取实际代码并测试
- 不修改业务代码

## 5. 标准工作流程

### 5.1 单任务执行流程

```
┌─────────────────┐
│   主 Agent      │
│ 选择下一个任务   │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  开发 Agent     │
│ 1. 读取任务     │
│ 2. 编写代码     │
│ 3. 自测         │
│ 4. 写开发报告   │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  测试 Agent     │
│ 1. 读取标准     │
│ 2. 独立验收     │
│ 3. 写测试报告   │
└────────┬────────┘
         │
         ▼
    ┌────────┐
    │ 通过?  │
    └───┬────┘
        │
   ┌────┴────┐
   │         │
  是        否
   │         │
   ▼         ▼
┌──────┐  ┌─────────────┐
│更新  │  │ 进入修正循环 │
│状态  │  │ (resume原开发)│
│下一个│  │   最多3轮    │
│任务  │  └─────────────┘
└──────┘
```

### 5.2 核心规则

**Developer 完成不等于任务完成**：
- Developer 完成 → 生成开发报告 → 等待 Tester 验收
- Tester 验收 → 生成测试报告 → 判定 PASS 或 FAIL
- **只有 Tester PASS 后，任务才算真正完成**

**修正循环规则**：
- 如果 Tester FAIL → 进入修正循环
- 最多 3 轮修正
- 谁写的 bug 谁修复（resume 原 Developer）
- 谁提出的 bug 谁复测（resume 原 Tester）

### 5.3 信息交接方式

子 Agent 有独立上下文，但共享项目文件系统。Agent 之间通过以下方式交接信息：

- `docs/tasks.md` - 任务清单和描述
- `docs/acceptance.md` - 验收标准
- `reports/dev/` - 开发报告
- `reports/test/` - 测试报告
- `memory/lessons.md` - 正向经验
- `memory/pitfalls.md` - 踩坑风险
- `worklog.md` - 整体协作过程记录

## 8. 关键经验

### 8.1 协作经验

**文档交接是关键**：
- 子 Agent 有独立上下文，不能依赖彼此记忆
- 项目文件系统是 Agent 之间的共享工作区
- Agent 之间通过 docs/、reports/、memory/ 交接信息

**职责清晰是基础**：
- 主 Agent 只负责调度和决策，不直接编码、不直接测试
- Developer 只负责一个明确任务，必须遵守最小必要修改
- Tester 必须独立验收，不能盲信 Developer 报告

**闭环修正是保障**：
- Developer 完成不等于任务完成
- 只有 Tester PASS 后，任务才算真正完成
- 如果 Tester FAIL，应进入修正循环
- 修正循环必须限制最大轮次（当前规则是最多 3 轮）

### 8.2 开发经验

**最小必要修改**：
- 只做当前任务需要的内容
- 不擅自扩展功能
- 不提前实现后续任务

**重构任务要小心**：
- 重构不得改变历史功能行为
- 对"用户输入"保持原值，不要擅自使用 trim()、toLowerCase()
- 重构任务必须做历史功能回归检查

### 8.3 测试经验

**独立验收必须有**：
- 不能只看开发报告就 PASS
- 必须读取实际代码并逐项对照验收标准
- Task 012 证明了 Tester 独立验收的价值

**发现问题要详细**：
- FAIL 时必须详细说明问题、复现方式、影响范围、建议修复
- 详细的反馈可以帮助 Developer 快速定位和修复问题

## 9. 风险与防护

### 9.1 已识别的风险

详见 `memory/pitfalls.md`，主要包括：

- **Agent 协作风险**：信息断层、职责混乱
- **Developer 风险**：一次改太多、擅自扩展功能、修改不应修改的文件
- **Tester 风险**：只看报告不读代码、漏掉行为变化
- **修正循环风险**：无限循环、责任链混乱
- **重构任务风险**：无意中改变历史功能行为

### 9.2 防护措施

**文档规范**：
- `docs/workflow.md` - 核心流程规范（所有 Agent 遵循）
- `.claude/agents/` - 子 Agent 模板（明确职责和边界）
- `memory/pitfalls.md` - 已知陷阱和避免方法

**流程控制**：
- 最多 3 轮修正循环
- 谁写的 bug 谁修复
- 谁提出的 bug 谁复测

**质量保证**：
- Tester 独立验收
- 重构任务必须做回归测试
- 详细的开发报告和测试报告

## 11. 如何运行 Todo Demo

### 方法 1：直接打开（推荐）

1. 找到项目根目录的 `index.html`
2. 双击文件，用浏览器打开

### 方法 2：使用 Python 本地服务器（可选）

```bash
python -m http.server 8000
```

然后在浏览器访问：

```
http://localhost:8000
```

### Todo 功能说明

**添加任务**：
- 在输入框中输入任务内容
- 点击"添加"按钮或按 Enter 键

**删除任务**：
- 点击任务右侧的 × 按钮

**切换完成状态**：
- 点击任务文本区域（不是删除按钮）

**注意事项**：
- 不能添加空任务或纯空格任务
- 列表为空时会显示友好提示
