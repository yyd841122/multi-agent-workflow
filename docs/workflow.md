# 多 Agent 协作开发流程规范

## 1. 流程目标

本规范定义 Claude Code 多 Agent 协作开发流程，确保：

- **明确分工**: 每个 Agent 职责清晰，避免职责混乱
- **独立验收**: 开发与测试分离，确保质量
- **闭环修复**: 测试失败后有明确修复流程
- **可追溯**: 所有过程有文档记录，便于复盘
- **可复用**: 流程模式可应用到其他项目

---

## 2. Agent 角色分工

### 2.1 主 Agent (Coordinator)

**职责**: 任务调度、流程控制、决策协调

**核心任务**:
1. 解析任务列表，选择下一个可执行任务
2. 将任务分配给开发 Agent
3. 协调测试 Agent 进行验收
4. 根据测试结果决定下一步行动（进入下一个任务 或 进入修正循环）
5. 更新 `docs/tasks.md` 中的任务状态
6. 更新 `worklog.md` 记录任务执行情况
7. 所有任务完成后生成项目总结

**输出格式**:
```
[任务分配消息]
```

**关键原则**:
- 不直接编码
- 不直接测试
- 只做调度和决策

---

### 2.2 计划 Agent (Planner)

**职责**: 任务拆分、依赖分析、验收标准制定

**核心任务**:
1. 读取 `docs/requirements.md` 需求文档
2. 将需求拆分为可独立完成的小任务（每个任务 30 分钟内完成）
3. 定义任务依赖关系
4. 制定详细验收标准
5. 评估任务优先级（P0/P1/P2/P3）

**输入文件**:
- `docs/requirements.md`

**输出文件**:
- `docs/plan.md` - 实施计划
- `docs/tasks.md` - 任务清单
- `docs/acceptance.md` - 验收标准

**输出格式**:
```
PLAN_FILE=docs/plan.md
TASKS_FILE=docs/tasks.md
ACCEPTANCE_FILE=docs/acceptance.md
STATUS=done
```

**关键原则**:
- 不编码
- 不测试
- 不创建业务代码文件

---

### 2.3 开发 Agent (Developer)

**职责**: 独立完成分配的开发任务

**核心任务**:
1. 读取任务文档（`docs/tasks.md`、`docs/acceptance.md`）
2. 读取相关业务代码
3. 进行最小必要修改，只做当前任务需要的内容
4. 进行自测（基本功能验证）
5. 编写开发报告（`reports/dev/task-xxx-dev-report.md`）
6. 在开发报告中记录 DEV_ID

**输入文件**:
- `docs/requirements.md`
- `docs/plan.md`
- `docs/tasks.md`
- `docs/acceptance.md`
- `memory/lessons.md`
- `memory/pitfalls.md`
- 业务代码文件

**输出文件**:
- 修改后的业务代码
- `reports/dev/task-xxx-dev-report.md`

**输出格式**:
```
DEV_REPORT=reports/dev/task-xxx-dev-report.md
STATUS=done
```

**关键原则**:
- 只处理主 Agent 分配的一个任务
- 不擅自处理其他任务
- 不一次性完成整个项目
- **不修改 `docs/tasks.md` 的任务状态**（由主 Agent 负责）
- 不修改验收文档

---

### 2.4 测试 Agent (Tester)

**职责**: 独立验收开发成果

**核心任务**:
1. 读取验收标准（`docs/acceptance.md`）
2. 读取开发报告（`reports/dev/task-xxx-dev-report.md`）
3. 根据验收标准逐项测试
4. 记录测试结果
5. 编写测试报告（`reports/test/task-xxx-test-report.md`）
6. 在测试报告中记录 TEST_ID

**输入文件**:
- `docs/requirements.md`
- `docs/tasks.md`
- `docs/acceptance.md`
- `reports/dev/task-xxx-dev-report.md`
- 业务代码文件

**输出文件**:
- `reports/test/task-xxx-test-report.md`

**输出格式**:
```
TEST_REPORT=reports/test/task-xxx-test-report.md
RESULT=pass
```
或
```
TEST_REPORT=reports/test/task-xxx-test-report.md
RESULT=fail
REASON=[失败原因简述]
```

**关键原则**:
- 只测试主 Agent 指定的一个任务
- 不修改业务代码
- **不修改 `docs/tasks.md` 的任务状态**（由主 Agent 负责）
- 提供详细的测试反馈
- 谁提出的 bug 谁复测

---

## 3. 单任务执行流程

### 3.1 流程图

```
┌─────────────────┐
│   主 Agent      │
│ 选择下一个任务   │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  开发 Agent     │
│ 1. 读取任务     │
│ 2. 编写代码     │
│ 3. 自测         │
│ 4. 写开发报告   │
│ 5. 记录 DEV_ID  │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  测试 Agent     │
│ 1. 读取标准     │
│ 2. 独立验收     │
│ 3. 写测试报告   │
│ 4. 记录 TEST_ID │
└────────┬────────┘
         │
         ▼
    ┌────────┐
    │ 通过?  │
    └───┬────┘
        │
   ┌────┴────┐
   │         │
  是        否
   │         │
   ▼         ▼
┌──────┐  ┌─────────────┐
│更新  │  │ 进入修正循环 │
│状态  │  │ (resume原开发)│
│下一个│  │   最多3轮    │
│任务  │  └─────────────┘
└──────┘
```

### 3.2 详细步骤

#### 步骤 1: 主 Agent 选择任务

1. 检查 `docs/tasks.md` 中的任务列表
2. 选择状态为 `pending` 且无未完成依赖的任务
3. 将任务分配给开发 Agent

#### 步骤 2: 开发 Agent 执行

1. 读取任务文档和验收标准
2. 读取相关业务代码
3. 进行最小必要修改
4. 进行自测
5. 编写开发报告（含 DEV_ID）
6. 返回简洁格式给主 Agent

#### 步骤 3: 测试 Agent 验收

1. 读取验收标准和开发报告
2. 逐项测试验收标准
3. 记录测试结果
4. 编写测试报告（含 TEST_ID）
5. 返回简洁格式给主 Agent

#### 步骤 4: 主 Agent 决策

- **测试通过**:
  - 更新 `docs/tasks.md` 状态为 `completed`
  - 更新 `worklog.md`
  - 进入下一个任务

- **测试失败**:
  - 进入修正循环

---

## 4. 修正循环规则

### 4.1 触发条件

测试 Agent 返回 `RESULT=fail`

### 4.2 修正流程

```
测试失败
   │
   ▼
主 Agent 通知原开发 Agent（通过 DEV_ID resume）
   │
   ▼
开发 Agent 阅读测试报告
   │
   ▼
开发 Agent 修复问题
   │
   ▼
开发 Agent 更新开发报告（追加修复记录）
   │
   ▼
原测试 Agent（通过 TEST_ID resume）重新验收
   │
   ▼
   ├─ 通过 → 结束循环，沉淀经验（如有）
   └─ 失败 → 计数+1，继续下一轮
```

### 4.3 关键规则

1. **谁写的 bug 谁修复**: 必须回到原开发 Agent 修复
2. **谁提出的 bug 谁复测**: 必须由原测试 Agent 复测
3. **最多 3 轮**:
   - 第 1 轮修复后仍失败 → 第 2 轮
   - 第 2 轮修复后仍失败 → 第 3 轮
   - 第 3 轮修复后仍失败 → 主 Agent 介入处理

### 4.4 修复报告

开发 Agent 每次修复后，需更新开发报告，在末尾追加：

```markdown
## 修复记录

### 第 N 轮修复
**修复时间**: [时间]
**问题描述**: [来自测试报告的问题]
**修复内容**: [如何修复的]
**验证方式**: [如何验证修复有效]
```

### 4.5 修正循环后的经验沉淀

修正循环结束后（无论成功还是主 Agent 介入），必须判断是否需要沉淀经验：

- 如果是因为**遗漏需求**导致的失败 → 沉淀到 `memory/pitfalls.md`
- 如果是因为**重构破坏历史功能**导致的失败 → 沉淀到 `memory/pitfalls.md`
- 如果形成了**可复用的修复方法** → 沉淀到 `memory/lessons.md`

## 5. Agent ID 记录规则

### 5.1 记录时机

- **开发报告**: 每次开发 Agent 完成任务后，必须在开发报告中记录 DEV_ID
- **测试报告**: 每次测试 Agent 完成验收后，必须在测试报告中记录 TEST_ID

### 5.2 ID 格式

在报告的"任务信息"或"测试信息"章节中添加：

**开发报告**:
```markdown
## 任务信息
- **任务 ID**: T001
- **任务名称**: 创建项目基础文件结构
- **开发 Agent**: Developer
- **DEV_ID**: [实际的 Agent Session ID 或 UNKNOWN]
- **开始时间**: 2026-04-28 13:56:00
- **完成时间**: 2026-04-28 13:56:00
```

**测试报告**:
```markdown
## 测试信息
- **任务 ID**: T001
- **任务名称**: 创建项目基础文件结构
- **测试 Agent**: Tester Agent
- **TEST_ID**: [实际的 Agent Session ID 或 UNKNOWN]
- **测试时间**: 2026-04-28
- **开发报告**: `reports/dev/task-001-dev-report.md`
```

### 5.3 获取规则

1. **优先获取真实 ID**: 如果当前环境支持获取 Agent Session ID，应记录真实 ID
2. **无法获取时记录 UNKNOWN**: 如果当前环境无法获取真实 Agent ID，先记录 `UNKNOWN`
3. **不要强行写死获取命令**: 不要在规范中写死未经验证的 Agent ID 获取命令，因为不同环境可能有不同的 API

### 5.4 修正循环中的 ID 使用

1. **优先 resume 原 Developer**: 进入修正循环时，主 Agent 应优先通过 DEV_ID resume 原开发 Agent
2. **优先 resume 原 Tester**: 修复后的复测，主 Agent 应优先通过 TEST_ID resume 原测试 Agent
3. **无法 resume 时报告**: 如果无法 resume 原 Agent，需要暂停并向主 Agent 报告，由主 Agent 决定后续处理方案

## 6. memory 经验沉淀规则

### 6.1 沉淀时机

**必须沉淀到 `memory/pitfalls.md`**:
- 每次出现 bug 导致返工
- 每次误改业务代码
- 每次遗漏需求细节
- 重构引发的历史功能破坏
- 修正循环失败（主 Agent 介入）

**必须沉淀到 `memory/lessons.md`**:
- 每次形成可复用的成功经验
- 每次发现可以标准化的流程
- 每次有效的测试方法
- 每次好的代码组织方式

### 6.2 pitfalls.md 格式

```markdown
## [问题描述]

### 陷阱描述
[详细描述这个陷阱是什么]

### 触发场景
[什么情况下会触发这个陷阱]

### 错误示例
```
[错误代码示例]
```

### 正确做法
```
[正确代码示例]
```

### 相关任务
- 任务 ID: [Txxx]
- 发生时间: [日期]

### 避免方法
[如何避免再次遇到这个陷阱]
```

### 6.3 lessons.md 格式

```markdown
## [经验标题]

### 经验描述
[这个经验是什么，解决了什么问题]

### 适用场景
[什么情况下可以使用这个经验]

### 实施步骤
1. [第一步]
2. [第二步]
3. [第三步]

### 相关任务
- 任务 ID: [Txxx]
- 形成时间: [日期]

### 注意事项
[使用这个经验时需要注意什么]
```

### 6.4 Task 012 经验示例（必须记录）

**记录到 `memory/pitfalls.md`**:

## 重构不能擅自改变历史功能行为

### 陷阱描述
重构代码时，不能擅自改变原有的功能行为，即使看起来是"优化"。

### 触发场景
- 进行代码重构任务时
- 提取常量或函数时
- 优化代码逻辑时

### 错误示例
例如 Task 012 中，重构时将 `taskInput.value` 改为 `taskInput.value.trim()`，导致历史输入保留行为被破坏。Task 005 的原有设计是保留用户输入的原始内容（包括空格），`trim()` 只用于空输入校验判断。重构后用户输入的首尾空格被丢弃，改变了历史功能行为。

### 正确做法
重构前先理解原有功能的完整行为，重构时只改代码结构、不改功能行为，对"用户输入"保持原值，不要擅自使用 trim()、toLowerCase() 等会改变原始值的方法。

### 相关任务
- 任务 ID: T012
- 发生时间: 2026-04-29

### 避免方法
1. 重构前先理解原有功能的完整行为
2. 重构时只改代码结构，不改功能行为
3. 重构后必须进行完整的回归测试
4. 完整的代码案例记录在 `memory/pitfalls.md` 本文件中

### 6.5 谁负责沉淀

- **修正循环结束后**: 由主 Agent 判断是否需要沉淀，如果需要则写入相应 memory 文件
- **普通任务完成后**: 如果 Agent 自己发现可复用经验，可以记录并告知主 Agent

## 7. skill 候选记录规则

### 7.1 记录时机

当以下内容可以复用时，应记录到 `memory/skills-candidates.md`：

- 标准化的流程（如：任务分配流程、验收流程）
- 检查清单（如：开发检查清单、测试检查清单）
- 报告模板（如：开发报告模板、测试报告模板）
- 修正循环方法
- 代码模式（如：事件委托模式、常量提取模式）

### 7.2 记录格式

```markdown
## [候选名称]

### 类型
- [ ] 流程类
- [ ] 检查清单类
- [ ] 报告模板类
- [ ] 代码模式类
- [ ] 其他

### 描述
[这个候选 skill 是什么，解决什么问题]

### 适用场景
[什么情况下可以使用这个 skill]

### 实施内容
```javascript
// 或 markdown 格式
[具体的实施步骤或代码]
```

### 来源
- 相关任务: [Txxx]
- 形成时间: [日期]
- 来源 Agent: [Developer/Tester/Planner]

### 复用价值评估
- [ ] 高 - 多项目通用
- [ ] 中 - 同类项目可用
- [ ] 低 - 特定场景可用
```

### 7.3 当前阶段策略

1. **只记录候选**: 当前阶段只记录到 `memory/skills-candidates.md`，不立即实现真正的 skill
2. **后续统一评估**: 项目完成后，统一评估哪些候选可以升级成 Claude Code skill
3. **不要过早实现**: 避免在流程验证阶段就投入大量精力实现 skill

## 8. 日志和报告规则

### 8.1 开发报告

**文件路径**: `reports/dev/task-xxx-dev-report.md`

**必要章节**:
1. 任务信息（任务 ID、任务名称、开发 Agent、DEV_ID、时间）
2. 任务描述
3. 完成内容（修改了哪些文件、做了什么）
4. 自测结果
5. 技术要点
6. 遗留问题（如有）
7. 备注（可选）
8. 修复记录（如有，修正循环时追加）

### 8.2 测试报告

**文件路径**: `reports/test/task-xxx-test-report.md`

**必要章节**:
1. 测试信息（任务 ID、测试 Agent、TEST_ID、测试时间、开发报告路径）
2. 验收结果（通过/失败）
3. 验收标准测试详情
4. 发现的问题（如有）
5. 开发报告核对
6. 测试结论
7. 下一步建议

### 8.3 工作日志

**文件路径**: `worklog.md`

**更新时机**: 每个任务完成后由主 Agent 更新

**记录内容**:
- 任务 ID、任务名称
- 状态、优先级、负责人
- 执行过程
- 交付成果
- 测试结果
- 验收标准检查项
- 前序任务保持情况
- 技术亮点（可选）

### 8.4 任务状态更新

**文件路径**: `docs/tasks.md`

**更新时机**: 任务测试通过后由主 Agent 更新

**状态值**:
- `pending` - 待分配
- `in_progress` - 开发中
- `completed` - 已完成

**注意**: 开发 Agent 和测试 Agent 都**不应该**修改此文件

## 9. 子 Agent 输出给主 Agent 的格式

### 9.1 计划 Agent 输出

```
PLAN_FILE=docs/plan.md
TASKS_FILE=docs/tasks.md
ACCEPTANCE_FILE=docs/acceptance.md
STATUS=done
```

**说明**:
- 只返回文件路径和完成状态
- 不返回大段计划内容
- 不返回大段解释

### 9.2 开发 Agent 输出

**成功完成**:
```
DEV_REPORT=reports/dev/task-xxx-dev-report.md
STATUS=done
```

**遇到阻塞**:
```
DEV_REPORT=reports/dev/task-xxx-dev-report.md
STATUS=blocked
REASON=[阻塞原因]
```

**说明**:
- 只返回报告路径和状态
- 不返回大段代码
- 不返回大段解释
- 详细内容写入开发报告

### 9.3 测试 Agent 输出

**测试通过**:
```
TEST_REPORT=reports/test/task-xxx-test-report.md
RESULT=pass
```

**测试失败**:
```
TEST_REPORT=reports/test/task-xxx-test-report.md
RESULT=fail
REASON=[失败原因简述，不超过 50 字]
```

**说明**:
- 只返回报告路径和结果
- 详细测试内容写入测试报告
- 失败时提供简要原因，便于主 Agent 快速了解问题

## 10. 项目总结汇报规则

### 10.1 总结时机

所有任务完成后（`docs/tasks.md` 中所有任务状态为 `completed`），主 Agent 需要生成项目级总结。

### 10.2 总结内容

**必选内容**:
1. **完成任务统计**
   - 总任务数
   - 按优先级分类统计（P0/P1/P2/P3）
   - 任务完成率

2. **修正循环统计**
   - 进入修正循环的任务数
   - 修正轮次统计（1 轮 / 2 轮 / 3 轮 / 主 Agent 介入）
   - 修正成功率

3. **主要修改文件清单**
   - 列出所有被修改的业务文件
   - 列出所有被创建的文档文件

4. **报告清单**
   - 开发报告数量和路径
   - 测试报告数量和路径

5. **经验沉淀**
   - `memory/lessons.md` 新增条目数
   - `memory/pitfalls.md` 新增条目数
   - `memory/skills-candidates.md` 新增候选数

6. **遗留问题**
   - 未解决的 Bug
   - 未实现的功能（如有）
   - 技术债务（如有）

**可选内容**:
7. **协作流程评估**
   - 任务粒度是否合理
   - Agent 分工是否清晰
   - 验收标准是否明确
   - 修正循环是否有效

8. **改进建议**
   - 流程改进建议
   - 工具改进建议
   - 文档改进建议

### 10.3 总结输出位置

**默认位置**: `worklog.md` 的"项目总结"部分

**复杂项目**: 如果项目较复杂（任务数 > 20 或修正循环次数 > 5），可以新增独立的 `reports/project-summary.md`

### 10.4 谁负责生成总结

由**主 Agent** 负责生成项目总结，因为：

1. 主 Agent 掌握全局视角
2. 主 Agent 负责调度所有任务
3. 主 Agent 处理所有修正循环
4. 主 Agent 统一更新所有状态

## 11. 质量保证机制

### 11.1 开发质量保证

- 开发前必须阅读相关文档
- 只做最小必要修改
- 不提前实现后续功能
- 完成后必须自测
- 自测结果写入开发报告
- 在开发报告中记录 DEV_ID

### 11.2 测试质量保证

- 测试前必须阅读验收标准
- 逐项测试所有验收标准
- 不遗漏任何验收点
- 失败时提供详细反馈
- 通过时也要核对开发报告
- 在测试报告中记录 TEST_ID

### 11.3 流程质量保证

- 每个任务都有完整文档（开发报告 + 测试报告）
- 每个任务都记录到 worklog.md
- 修正循环有完整记录
- 主 Agent 统一管理任务状态
- 经验及时沉淀到 memory

## 12. 成功标准

### 12.1 任务成功标准

- 所有验收标准通过
- 测试报告标记为 `RESULT=pass`
- 开发报告与测试报告一致
- 无遗留严重问题

### 12.2 流程成功标准

- 至少完成 3 个完整的开发-验收循环
- 至少经历 1 次修正循环（可选，用于验证修复流程）
- 所有报告完整且格式统一
- worklog.md 记录完整
- 任务状态更新及时
- memory 经验沉淀完整
