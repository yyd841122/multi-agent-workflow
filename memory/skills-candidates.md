# Skill 候选库

本文档记录在多 Agent 协作开发流程中发现的、可能抽象为 Claude Code Skill 的候选项。

---

## 1. 文档目的

### 1.1 为什么需要 Skill 候选库

在多 Agent 协作开发过程中，我们发现一些流程、检查清单、报告模板具有高度复用性。将这些内容抽象为 Skill 可以：

- **减少重复工作**: 避免每次都手动执行相同的流程
- **标准化协作**: 确保每个项目遵循相同的流程规范
- **降低出错率**: 通过 Skill 固化最佳实践
- **提升效率**: 自动化繁琐的流程控制

### 1.2 当前阶段的策略

**只记录候选，不立即实现**

- 当前阶段只记录到本文档，作为 Skill 候选
- 不立即创建真正的 Claude Code Skill
- 经过多个项目验证后，再评估哪些候选值得实现
- 避免过早投入，确保 Skill 的实用性

### 1.3 候选记录时机

当以下内容可以复用时，应记录到本文档：

- 标准化的流程（如：任务分配流程、验收流程）
- 检查清单（如：开发检查清单、测试检查清单）
- 报告模板（如：开发报告模板、测试报告模板）
- 修正循环方法
- **服务于多 Agent 协作的代码实践**（如：可复用的测试方法、回归测试策略）

**不记录**：
- 普通的代码技巧或优化方法
- 与多 Agent 协作无关的技术细节
- 可以在通用编程资料中找到的知识

### 1.4 谁负责记录

- **Developer Agent**: 发现可复用的开发流程、协作模式时记录
- **Tester Agent**: 发现可复用的测试方法、检查清单时记录
- **主 Agent**: 发现可复用的协调流程、控制逻辑时记录

---

## 2. 记录规则

### 2.1 候选 Skill 格式

每个候选项按以下格式记录：

```markdown
### skill-name

**用途**: [这个 Skill 解决什么问题]

**适用场景**: [什么情况下可以使用这个 Skill]

**输入**: [需要什么输入参数]

**输出**: [产生什么输出结果]

**当前状态**: 候选 / 待验证 / 可优先实现

**备注**: [其他说明]
```

### 2.2 状态说明

- **候选**: 首次发现，尚未在其他项目验证
- **待验证**: 已在本项目验证，需要在更多项目验证
- **可优先实现**: 经过多个项目验证，值得优先实现为正式 Skill

### 2.3 记录原则

1. **只记录与多 Agent 协作相关**: 不记录无关的技术能力
2. **说明解决的问题**: 每个候选项都要明确它解决什么问题
3. **说明未来使用方式**: 每个候选项都要说明未来可能如何使用
4. **保持简洁**: 不记录详细的实现细节，只记录核心思路

**重要限制**：

- **代码模式类内容限制**: 只有在能服务于多 Agent 协作、测试、回归、报告生成时，才记录为 Skill 候选
- **普通代码技巧暂不记录**: 避免本库变成代码技巧收集库
- **聚焦协作流程**: 本库专注于多 Agent 协作流程的优化，而非通用编程知识

## 3. 当前 Skill 候选列表

### 3.1 workflow-controller

**用途**: 控制主 Agent 按照固定的多 Agent 协作流程执行任务，避免跳步或遗漏环节。

**适用场景**:
- 主 Agent 需要协调多个子 Agent 完成开发任务
- 需要遵循"开发 → 测试 → 修复 → 复测"的固定流程
- 需要确保每个环节都有文档记录

**输入**:
- `docs/tasks.md` - 任务清单
- `docs/acceptance.md` - 验收标准
- 当前任务 ID

**输出**:
- 任务分配消息
- 状态更新决策
- 下一步行动指令

**核心流程**:
```
选择任务 → 分配给 Developer → 等待开发完成 →
分配给 Tester → 等待测试结果 →
  ├─ PASS → 更新状态 → 下一个任务
  └─ FAIL → 通知原 Developer 修复 → 最多 3 轮 →
      ├─ PASS → 更新状态 → 下一个任务
      └─ FAIL → 主 Agent 介入
```

**当前状态**: 候选

**备注**:
- 这是多 Agent 协作的核心控制器
- 本项目已验证流程可行（12 个任务，100% 通过率）
- 需要在更多项目验证不同场景下的适应性

---

### 3.2 task-report-generator

**用途**: 生成标准化的开发报告和测试报告，确保报告格式统一、信息完整。

**适用场景**:
- Developer Agent 完成任务后生成开发报告
- Tester Agent 完成验收后生成测试报告
- 需要确保报告包含所有必要章节

**输入**:
- 任务 ID
- 任务描述
- 完成内容
- 测试结果
- Agent ID（DEV_ID 或 TEST_ID）

**输出**:
- `reports/dev/task-xxx-dev-report.md`
- `reports/test/task-xxx-test-report.md`

**报告模板**:

**开发报告模板**:
```markdown
## 任务信息
- 任务 ID
- 任务名称
- 开发 Agent
- DEV_ID
- 开始时间
- 完成时间

## 任务描述
## 完成内容
## 自测结果
## 技术要点
## 遗留问题
## 备注
```

**测试报告模板**:
```markdown
## 测试信息
- 任务 ID
- 任务名称
- 测试 Agent
- TEST_ID
- 测试时间
- 开发报告路径

## 验收结果
- 总体结果: PASS / FAIL

## 验收标准测试详情
## 发现的问题
## 开发报告核对
## 测试结论
## 下一步建议
```

**当前状态**: 待验证

**备注**:
- 本项目已使用此模板生成了 12 个开发报告和 12 个测试报告
- 模板结构清晰，易于理解和维护
- 可以考虑进一步抽象为 Skill，自动填充部分内容

### 3.3 fix-loop-controller

**用途**: 控制测试失败后的修正循环，确保修复流程有序进行，最多 3 轮，避免无限循环。

**适用场景**:
- Tester Agent 返回 `RESULT=fail`
- 需要通知原 Developer Agent 修复
- 需要限制修正轮次，避免无限循环

**输入**:
- 测试报告（包含失败原因）
- 开发报告（包含 DEV_ID）
- 当前修正轮次计数

**输出**:
- 修复通知消息
- 修正轮次更新
- 继续修正 或 主 Agent介入的决策

**核心流程**:
```
测试失败 → 通过 DEV_ID resume 原 Developer →
Developer 修复 → 更新开发报告 →
通过 TEST_ID resume 原 Tester →
Tester 复测 →
  ├─ PASS → 结束修正循环
  └─ FAIL → 轮次+1 →
      ├─ 轮次 < 3 → 继续下一轮
      └─ 轮次 = 3 → 主 Agent 介入
```

**当前状态**: 待验证

**备注**:
- 本项目 Task 012 验证了修正循环流程（1 轮修复成功）
- 关键点：通过 DEV_ID 和 TEST_ID resume 同一个 Agent
- 需要验证更复杂的多轮修正场景

### 3.4 agent-id-recorder

**用途**: 记录 Developer 和 Tester 的 Agent Session ID、报告路径和任务编号，便于修正循环时 resume。

**适用场景**:
- 开发报告需要记录 DEV_ID
- 测试报告需要记录 TEST_ID
- 修正循环需要通过 ID resume 同一个 Agent

**输入**:
- Agent Session ID（如果能获取）
- 任务 ID
- 报告文件路径

**输出**:
- 在报告中添加 Agent ID 字段
- 维护 Agent ID 到任务和报告的映射关系

**记录格式**:

**开发报告**:
```markdown
## 任务信息
- **DEV_ID**: [实际的 Agent Session ID 或 UNKNOWN]
```

**测试报告**:
```markdown
## 测试信息
- **TEST_ID**: [实际的 Agent Session ID 或 UNKNOWN]
```

**当前状态**: 候选

**备注**:
- 当前环境可能无法获取真实 Agent ID，先记录 UNKNOWN
- 这是修正循环能够正常工作的关键基础设施
- 需要验证不同环境下的 Agent ID 获取方法

### 3.5 regression-test-checklist

**用途**: 根据历史任务和验收标准，自动生成回归测试清单，确保重构类任务不破坏历史功能。

**适用场景**:
- 重构类任务（代码重构、样式重构）
- 修改核心逻辑的任务
- 修改公共函数的任务
- 需要验证历史功能是否被破坏

**输入**:
- 当前任务 ID（如 T012）
- `docs/tasks.md` - 历史任务列表
- `docs/acceptance.md` - 历史验收标准
- 当前任务的修改范围

**输出**:
- 回归测试清单
- 需要重点测试的功能列表

**检查清单示例**:

```markdown
## 回归测试清单

### 基础功能测试
- [ ] 页面能否正常打开
- [ ] 能否添加任务
- [ ] 能否删除任务
- [ ] 能否切换完成状态
- [ ] 空输入校验是否正常

### 行为保持测试
- [ ] 输入包含空格，保存时是否保留空格
- [ ] 所有历史功能行为是否改变
```

**当前状态**: 待验证

**备注**:
- 本项目 Task 012 发现了重构破坏历史功能的问题
- 回归测试是防止此类问题的关键
- 可以进一步抽象为自动生成检查清单的 Skill

### 3.6 memory-updater

**用途**: 将开发过程中发现的、服务于多 Agent 协作的经验和方法，自动沉淀到 memory/lessons.md 和 memory/pitfalls.md。

**适用场景**:
- Developer Agent 发现可复用的协作流程
- Tester Agent 发现容易犯错的地方
- 修正循环结束后总结经验
- 发现可以优化的协作方式

**输入**:
- 经验类型（lesson / pitfall）
- 经验描述
- 相关任务 ID
- 形成时间

**输出**:
- 更新 `memory/lessons.md`
- 更新 `memory/pitfalls.md`

**记录格式**:

**lessons.md**:
```markdown
## [经验标题]

### 经验描述
[这个经验是什么，解决了什么问题]

### 适用场景
[什么情况下可以使用这个经验]

### 实施步骤
1. [第一步]
2. [第二步]

### 相关任务
- 任务 ID: [Txxx]
- 形成时间: [日期]
```

**pitfalls.md**:
```markdown
## [问题描述]

### 陷阱描述
[详细描述这个陷阱是什么]

### 触发场景
[什么情况下会触发这个陷阱]

### 错误示例
[错误代码示例]

### 正确做法
[正确代码示例]

### 相关任务
- 任务 ID: [Txxx]
- 发生时间: [日期]
```

**当前状态**: 候选

**备注**:
- 当前阶段为手动记录
- 未来可以自动识别需要沉淀的经验
- 关键是要有明确的判断标准
- **只记录服务于多 Agent 协作的经验**，不记录通用编程技巧

### 3.7 project-summary-generator

**用途**: 项目完成后自动生成项目级总结汇报，包括任务统计、修正循环统计、文件清单、经验沉淀等。

**适用场景**:
- 所有任务完成后（`docs/tasks.md` 中所有任务状态为 `completed`）
- 需要生成项目总结报告
- 需要统计协作成果

**输入**:
- `docs/tasks.md` - 任务列表
- `reports/dev/` - 所有开发报告
- `reports/test/` - 所有测试报告
- `worklog.md` - 工作日志
- `memory/` - 经验库

**输出**:
- 项目总结（写入 `worklog.md` 或 `reports/project-summary.md`）

**总结内容**:

**必选内容**:
1. 完成任务统计（总任务数、按优先级分类、完成率）
2. 修正循环统计（进入修正循环的任务数、轮次、成功率）
3. 主要修改文件清单
4. 报告清单（开发报告、测试报告）
5. 经验沉淀（lessons、pitfalls、skills-candidates）
6. 遗留问题

**可选内容**:
7. 协作流程评估
8. 改进建议

**当前状态**: 候选

**备注**:
- 本项目已完成 12 个任务，适合验证此 Skill
- 总结可以写入 `worklog.md` 的"项目总结"部分
- 复杂项目可以生成独立的 `reports/project-summary.md`

## 4. 候选 Skill 评估标准

### 4.1 何时升级为正式 Skill

**必须满足**：
1. **经过多个项目验证**: 至少在 3 个以上项目中使用
2. **证明有明显价值**: 能显著提升效率或降低出错率
3. **实现难度可控**: 技术上可行，不会过于复杂

**加分项**：
- 使用频率高（每次协作都用到）
- 社区有强烈需求
- 有明确的 ROI（投入产出比）

### 4.2 优先级判断

**可优先实现**:
- workflow-controller - 核心流程控制
- task-report-generator - 报告生成自动化

**可延后实现**:
- agent-id-recorder - 依赖环境支持
- memory-updater - 可以先手动维护

**需要更多验证**:
- fix-loop-controller - 需要更多修正循环案例
- regression-test-checklist - 需要更多重构任务
- project-summary-generator - 需要更多完整项目

- 分步实现：先实现最核心的
- 持续迭代：根据反馈不断优化
