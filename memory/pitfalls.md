# Pitfalls

本文档记录在多 Agent 协作开发过程中出现过的风险、错误模式和踩坑案例。

这些陷阱在实际项目中已经发生过，后续项目中需要注意避免。

---

## 文档目的

**memory/pitfalls.md 只记录错误细节、踩坑案例、风险模式、避免方法。已经验证有效的正向经验应记录到 `memory/lessons.md`。未来可能抽象成 Skill 的方法应记录到 `memory/skills-candidates.md`。**

三个 memory 文件的分工：
- **`memory/lessons.md`**：记录已经验证有效、后续值得复用的正向经验
- **`memory/pitfalls.md`**（本文档）：记录错误细节、踩坑案例、风险模式、避免方法
- **`memory/skills-candidates.md`**：记录未来可能抽象成 Skill 的候选流程和方法

## 1. Agent 协作风险

### 1.1 子 Agent 上下文独立，不能假设另一个 Agent 知道当前对话内容

**陷阱描述**：
每个子 Agent（Planner、Developer、Tester）都有独立的上下文，不能假设另一个 Agent 知道当前对话内容。如果没有文档交接，Agent 之间容易信息断层。

**触发场景**：
- Developer 认为 Planner 知道自己为什么这样实现
- Tester 认为 Developer 知道自己如何测试
- 主 Agent 认为子 Agent 记住了之前的讨论

**错误示例**：
Developer 在任务中做了一些优化，但没有在开发报告中记录，认为主 Agent 知道。

**正确做法**：
所有信息必须通过文件系统交接：
- Developer → 开发报告
- Tester → 测试报告
- 主 Agent 读取报告了解详情

**相关任务**：
- 任务 ID: T001-T012
- 发生时间: 2026-04-28 至 2026-04-29

**避免方法**：
- 在 Agent 模板中明确说明上下文独立
- 强调必须通过文件交接
- 不要假设其他 Agent 记住了什么

---

### 1.2 如果没有文档交接，Agent 之间容易信息断层

**陷阱描述**：
子 Agent 之间的信息传递必须通过文档，不能依赖口头传递或隐式假设。如果没有文档交接，信息容易断层或丢失。

**触发场景**：
- Developer 完成任务后不写开发报告
- Tester 发现问题后不写测试报告
- 主 Agent 只凭记忆更新状态

**错误示例**：
Developer 完成任务后只说"完成了"，但没有写开发报告，主 Agent 和 Tester 不知道具体做了什么。

**正确做法**：
每个环节都有明确的输入输出文件：
- Developer 必须生成开发报告
- Tester 必须生成测试报告
- 主 Agent 必须更新 worklog.md

**相关任务**：
- 任务 ID: T001-T012
- 发生时间: 2026-04-28 至 2026-04-29

**避免方法**：
- 在 Agent 模板中定义明确的输入输出文件
- 检查是否生成了对应的报告文件
- 报告文件路径必须在 Agent 输出中返回

## 2. 主 Agent 风险

### 2.1 主 Agent 不能直接替代 Developer 编码

**陷阱描述**：
主 Agent 应该专注于调度和决策，不应该直接编码。如果主 Agent 直接替代 Developer 编码，会破坏职责分工。

**触发场景**：
- 主 Agent 认为"简单的改一下就好"
- 主 Agent 认为"这样更快"
- Developer 遇到困难时，主 Agent 直接接手

**错误示例**：
主 Agent 看到 Developer 的代码，觉得有个地方不够好，直接修改了代码。

**正确做法**：
主 Agent 只负责：
- 选择任务并分配给 Developer
- 根据测试结果决定下一步
- 更新任务状态和 worklog.md

**相关任务**：
- 任务 ID: T001-T012
- 发生时间: 2026-04-28 至 2026-04-29

**避免方法**：
- 在主 Agent 规范中明确禁止直接编码
- 主 Agent 想修改代码时，应该分配给 Developer

---

### 2.2 主 Agent 不能直接替代 Tester 验收

**陷阱描述**：
主 Agent 应该专注于调度和决策，不应该直接测试验收。如果主 Agent 直接替代 Tester 验收，会破坏质量把关。

**触发场景**：
- 主 Agent 认为"看起来没问题"
- 主 Agent 认为"不用那么麻烦"
- Tester 遇到困难时，主 Agent 直接接手

**错误示例**：
主 Agent 看了 Developer 的代码，觉得没问题，直接更新状态为 completed，没有分配给 Tester。

**正确做法**：
主 Agent 只负责：
- 将任务分配给 Tester 验收
- 根据测试报告的判定结果决策
- 不要自己判断测试通过与否

**相关任务**：
- 任务 ID: T001-T012
- 发生时间: 2026-04-28 至 2026-04-29

**避免方法**：
- 在主 Agent 规范中明确禁止直接测试
- 主 Agent 必须等待 Tester 的测试报告

## 3. Developer 常见风险

### 3.1 Developer 容易一次改太多，超出当前任务范围

**陷阱描述**：
Developer 容易一次性做太多内容，超出当前任务的范围。这会导致：
- 提前实现了后续任务的功能
- 增加了测试难度
- 破坏了任务拆分的逻辑

**触发场景**：
- 任务之间有关联时
- Developer 认为"反正迟早要做"
- Developer 认为"这样更好"

**错误示例**：
任务要求"添加输入框"，Developer 却同时添加了按钮、列表和样式。

**正确做法**：
只做当前任务需要的内容：
- 仔细阅读验收标准
- 只做验收标准中明确列出的内容
- 验收标准没有提到的功能，坚决不做

**相关任务**：
- 任务 ID: T001-T011
- 发生时间: 2026-04-28 至 2026-04-29

**避免方法**：
- 在 Agent 模板中强调"最小必要修改"
- 在 memory/lessons.md 中记录正确做法
- Tester 发现超出范围时必须在测试报告中指出

---

### 3.2 Developer 可能擅自扩展功能

**陷阱描述**：
Developer 可能擅自添加"锦上添花"的功能，即使验收标准中没有要求。这会破坏任务边界的清晰性。

**触发场景**：
- Developer 认为某个功能"很有用"
- Developer 认为某个改动"很简单"
- Developer 想展示自己的能力

**错误示例**：
任务要求"实现添加任务功能"，Developer 还顺便实现了"任务编辑功能"，认为"这样更完整"。

**正确做法**：
验收标准写什么，就做什么：
- 没写的，不做
- 后续任务的功能，不提前做
- "锦上添花"的功能，不擅自添加

**相关任务**：
- 任务 ID: T001-T011
- 发生时间: 2026-04-28 至 2026-04-29

**避免方法**：
- 在 Agent 模板中明确禁止擅自扩展功能
- Tester 发现擅自扩展时必须在测试报告中指出

---

### 3.3 Developer 可能修改 docs/tasks.md 状态，这是禁止的

**陷阱描述**：
Developer 可能会修改 `docs/tasks.md` 的任务状态，但这是禁止的。任务状态应该由主 Agent 统一管理。

**触发场景**：
- Developer 完成任务后认为"应该更新状态"
- Developer 不知道这是禁止的
- Developer 认为"这样更方便"

**错误示例**：
Developer 完成任务后，将 `docs/tasks.md` 中的状态从 `pending` 改为 `completed`。

**正确做法**：
Developer 只负责：
- 修改业务代码
- 生成开发报告
- 返回简洁格式给主 Agent

主 Agent 负责：
- 读取测试报告
- 更新任务状态
- 更新 worklog.md

**相关任务**：
- 任务 ID: T001-T012
- 发生时间: 2026-04-28 至 2026-04-29

**避免方法**：
- 在 Developer Agent 模板中明确禁止修改 `docs/tasks.md`
- 在测试报告中检查任务状态是否被修改

---

### 3.4 Developer 可能修改 docs/acceptance.md 验收标准，这是禁止的

**陷阱描述**：
Developer 可能会修改 `docs/acceptance.md` 的验收标准，但这是禁止的。验收标准应该由 Planner Agent 制定，主 Agent 协调，不应该由 Developer 修改。

**触发场景**：
- Developer 认为验收标准"不合理"
- Developer 认为验收标准"不完整"
- Developer 想降低验收难度

**错误示例**：
Developer 发现某项验收标准很难实现，于是修改了 `docs/acceptance.md`，降低了验收要求。

**正确做法**：
- Developer 严格按照验收标准执行
- 如果验收标准有问题，应该在开发报告中说明
- 由主 Agent 决定是否需要调整验收标准

**相关任务**：
- 任务 ID: T001-T012
- 发生时间: 2026-04-28 至 2026-04-29

**避免方法**：
- 在 Developer Agent 模板中明确禁止修改 `docs/acceptance.md`
- 在测试报告中检查验收标准是否被修改

## 4. Tester 常见风险

### 4.1 Tester 不能只看 Developer 报告就 PASS

**陷阱描述**：
Tester 可能只看开发报告就认为没问题，但这样容易漏掉行为变化和 bug。必须读取实际代码并逐项对照验收标准。

**触发场景**：
- Tester 认为"Developer 说做了就是做了"
- Tester 嫌麻烦，不想看代码
- Tester 信任 Developer 的能力

**错误示例**：
Developer 在开发报告中说"实现了添加任务功能"，Tester 没有看代码就直接 PASS，实际上添加功能有 bug。

**正确做法**：
独立验收的步骤：
1. 读取开发报告，了解 Developer 声称做了什么
2. 读取业务代码，确认实际做了什么
3. 逐项对照验收标准，每项都要测试
4. 判定 PASS 或 FAIL

**相关任务**：
- 任务 ID: T012
- 发生时间: 2026-04-29

**避免方法**：
- 在 Tester Agent 模板中强调独立验收
- 明确说明"不能只看开发报告就 PASS"

---

### 4.2 Tester 必须读取实际代码，否则容易漏掉行为变化

**陷阱描述**：
Tester 可能只做功能测试，不读取实际代码。但这样容易漏掉代码中的行为变化，特别是在重构任务中。

**触发场景**：
- Tester 认为"功能正常就行"
- Tester 不想看代码，觉得麻烦
- Tester 信任 Developer 的实现

**错误示例**：
重构任务中，Developer 将 `taskInput.value` 改为 `taskInput.value.trim()`，Tester 只测试了"能添加任务"，没有检查代码，导致漏掉了"用户输入空格被丢弃"的行为变化。

**正确做法**：
Tester 必须：
1. 读取业务代码，确认实际做了什么
2. 检查代码是否有行为变化
3. 特别关注重构任务中的行为一致性
4. 逐项对照验收标准

**相关任务**：
- 任务 ID: T012
- 发生时间: 2026-04-29

**避免方法**：
- 在 Tester Agent 模板中明确要求读取代码
- 重构任务必须做回归测试

## 5. 修正循环风险

### 5.1 修正循环不能无限循环，最多 3 轮

**陷阱描述**：
如果没有修正循环的上限，可能会陷入无限循环。Developer 和 Tester 可能对某个问题反复争论和修改，无法收敛。

**触发场景**：
- Developer 和 Tester 对验收标准理解不一致
- 需求本身有问题，无法满足
- Developer 能力不足，无法修复

**错误示例**：
某任务经过了 5 轮修正循环，仍然无法通过，浪费了大量时间和资源。

**正确做法**：
设置修正循环上限：
- 最多 3 轮修正
- 第 3 轮失败后，主 Agent 介入
- 主 Agent 评估是否需要调整验收标准或需求

**相关任务**：
- 任务 ID: T012
- 发生时间: 2026-04-29

**避免方法**：
- 在 workflow.md 中明确修正循环上限
- 在 Agent 模板中说明修正循环规则

---

### 5.2 如果不是原 Developer 修 bug，责任链容易混乱

**陷阱描述**：
修正循环时，如果不是回到原 Developer Agent 修复，责任链容易混乱，新 Developer 可能不了解之前的上下文。

**触发场景**：
- 主 Agent 随意分配新的 Developer 修复
- 原 Developer 上下文已丢失，无法 resume
- 多个 Developer 参与修复，责任不清

**错误示例**：
Tester 发现 bug，主 Agent 分配给新的 Developer 修复，新 Developer 不了解之前的上下文，修复方向错误。

**正确做法**：
谁写的 bug 谁修复：
1. 主 Agent 通过 DEV_ID resume 原 Developer Agent
2. 原 Developer 阅读测试报告，修复问题
3. 原 Developer 更新开发报告，追加修复记录

**相关任务**：
- 任务 ID: T012
- 发生时间: 2026-04-29

**避免方法**：
- 在开发报告和测试报告中记录 Agent ID
- 修正循环时优先 resume 原 Agent

---

### 5.3 如果不是原 Tester 复测，bug 验收标准容易变化

**陷阱描述**：
修正循环时，如果不是回到原 Tester Agent 复测，bug 的验收标准容易变化，新 Tester 可能不了解之前的判定依据。

**触发场景**：
- 主 Agent 随意分配新的 Tester 复测
- 原 Tester 上下文已丢失，无法 resume
- 多个 Tester 参与验收，标准不一致

**错误示例**：
Tester 1 判定 FAIL，Developer 修复后，主 Agent 分配给 Tester 2 复测，Tester 2 改变了验收标准，判定 PASS。

**正确做法**：
谁提出的 bug 谁复测：
1. 主 Agent 通过 TEST_ID resume 原 Tester Agent
2. 原 Tester 阅读修复记录，复测问题
3. 确保验收标准一致

**相关任务**：
- 任务 ID: T012
- 发生时间: 2026-04-29

**避免方法**：
- 在开发报告和测试报告中记录 Agent ID
- 修正循环时优先 resume 原 Agent

## 6. 日志和报告风险

### 6.1 子 Agent 输出过长会污染主 Agent 上下文

**陷阱描述**：
子 Agent（Developer、Tester）返回给主 Agent 的内容如果过长，会污染主 Agent 的上下文，导致主 Agent 无法高效工作。

**触发场景**：
- Developer 返回大段代码
- Tester 返回详细的测试过程
- 子 Agent 返回大段解释

**错误示例**：
Developer 完成任务后，直接将所有修改的代码返回给主 Agent，导致主 Agent 上下文过载。

**正确做法**：
子 Agent 输出必须简洁：
- Developer 输出：`DEV_REPORT=reports/dev/task-xxx-dev-report.md` + `STATUS=done` + `DEV_ID`
- Tester 输出：`TEST_REPORT=reports/test/task-xxx-test-report.md` + `RESULT=pass/fail` + `TEST_ID`
- 详细内容写入报告文件

**相关任务**：
- 任务 ID: T001-T012
- 发生时间: 2026-04-28 至 2026-04-29

**避免方法**：
- 在 Agent 模板中定义明确的输出格式
- 强调详细内容写入报告

---

### 6.2 详细内容必须写入 reports/dev/ 和 reports/test/

**陷阱描述**：
如果子 Agent 不将详细内容写入报告文件，会导致信息丢失，后续无法追溯。

**触发场景**：
- Developer 只返回简洁格式，不写开发报告
- Tester 只返回简洁格式，不写测试报告
- 报告内容不完整，缺少关键信息

**错误示例**：
Developer 完成任务后，只返回 `STATUS=done`，没有生成开发报告，主 Agent 不知道具体做了什么。

**正确做法**：
必须生成完整的报告：
- `reports/dev/task-xxx-dev-report.md`：包含任务信息、完成内容、自测结果、技术要点等
- `reports/test/task-xxx-test-report.md`：包含测试信息、验收结果、测试详情、发现的问题等

**相关任务**：
- 任务 ID: T001-T012
- 发生时间: 2026-04-28 至 2026-04-29

**避免方法**：
- 在 Agent 模板中明确要求生成报告
- 检查报告文件是否存在

### 6.3 本地配置文件容易被自动修改，通常不要提交

**陷阱描述**：
`.claude/settings.local.json` 是本地配置文件，容易被 Claude Code 自动修改。如果提交这个文件，会导致其他开发者的配置被覆盖。

**触发场景**：
- Claude Code 自动修改了 settings.local.json
- 开发者误将这个文件提交到 git
- 其他开发者拉取后配置被覆盖

**错误示例**：
开发者执行 `git add .` 和 `git commit`，将 `.claude/settings.local.json` 提交到了 git。

**正确做法**：
1. 在 `.gitignore` 中添加 `.claude/settings.local.json`
2. 执行 `git add` 时明确指定文件，不要使用 `git add .`
3. 在 git status 中检查是否包含这个文件

**相关任务**：
- 任务 ID: T017
- 发生时间: 2026-04-30

**避免方法**：
- 在项目初始化时就配置 .gitignore
- 养成检查 git status 的习惯

## 7. 重构任务风险

### 7.1 重构任务必须进行历史功能回归检查

**陷阱描述**：
重构任务容易无意中改变历史功能行为。如果不进行完整的回归测试，会破坏已有功能。

**触发场景**：
- 代码重构（提取函数、优化逻辑）
- 样式重构（调整布局、修改配色）
- 修改核心逻辑或公共函数

**错误示例**：
Developer 重构了 `addTask()` 函数，提高了代码可读性，但无意中改变了输入校验的行为，导致历史功能被破坏。

**正确做法**：
重构任务必须做回归测试：
1. 重构前，列出所有相关历史功能
2. 重构时，只改代码结构，不改功能行为

例如：

```powershell
chcp 65001
[Console]::OutputEncoding = [System.Text.Encoding]::UTF8
Get-Content -Encoding UTF8 docs\workflow.md -TotalCount 40
```

## 10. 已验证踩坑案例

### 10.1 Task 012：重构时改变历史功能行为

**陷阱描述**：
重构时 Developer 可能在看似优化代码的过程中，无意中改变历史功能行为。Task 012 是一个典型的踩坑案例。

**触发场景**：
- 代码重构任务
- 提取常量或函数时
- 优化代码逻辑时

**原设计**：

```javascript
const taskText = taskInput.value;

if (taskText.trim() === '') {
    showInputError();
    return;
}

// 后续使用 taskText 添加到列表
```

**说明**：
`trim()` 只用于空输入校验判断，不能改变最终添加到列表中的有效输入内容。Task 005 的原有设计是保留用户输入的原始内容（包括空格）。

**错误风险**：
如果改成：

```javascript
const taskText = taskInput.value.trim();  // ❌ 错误

if (taskText === '') {
    showInputError();
    return;
}
```

用户输入首尾空格时，最终内容会被改变，破坏历史验收要求。例如用户输入 `"  buy milk  "`，会被保存为 `"buy milk"`，空格被丢弃了。

**正确做法**：

```javascript
const taskText = taskInput.value;  // ✅ 正确

if (taskText.trim() === '') {
    showInputError();
    return;
}

// 后续使用 taskText（原始值）添加到列表
```

**预防规则**：
1. 重构任务必须做历史功能回归检查
2. 重构时只改代码结构，不改功能行为
3. 对"用户输入"保持原值，不要擅自使用 trim()、toLowerCase() 等
4. Tester 必须独立验收，检查历史功能是否被改变
5. 特别关注行为一致性，不能只看功能是否可用

**相关任务**：
- 任务 ID: T012
- 发生时间: 2026-04-29

**避免方法**：
- 重构前先理解原有功能的完整行为
- 在 memory/pitfalls.md 中记录这个案例
- 在 Developer 和 Tester Agent 模板中强调这个风险
