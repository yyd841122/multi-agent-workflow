# Lessons Learned

本文档记录在多 Agent 协作开发过程中形成的、已经验证有效的经验。

这些经验经过实际项目验证（本项目 12 个任务，100% 通过率），值得在后续项目中复用。

---

## 文档目的

**memory/lessons.md 只记录已经验证有效、后续值得复用的正向经验。错误细节、踩坑案例、风险模式应记录到 `memory/pitfalls.md`。未来可能抽象成 Skill 的方法应记录到 `memory/skills-candidates.md`。**

三个 memory 文件的分工：
- **`memory/lessons.md`**（本文档）：记录已经验证有效、后续值得复用的正向经验
- **`memory/pitfalls.md`**：记录错误细节、踩坑案例、风险模式、避免方法
- **`memory/skills-candidates.md`**：记录未来可能抽象成 Skill 的候选流程和方法

## 1. 文档交接经验

### 1.1 子 Agent 通过文件系统交接，不依赖彼此记忆

**经验描述**：
每个子 Agent（Planner、Developer、Tester）都有独立的上下文，不能依赖彼此的记忆。必须通过文件系统进行信息交接，项目文件系统是 Agent 之间的共享工作区。

**适用场景**：
- 任何多 Agent 协作项目
- 需要在不同 Agent 之间传递信息时
- 需要持久化协作过程时

**实施步骤**：
1. Planner Agent 将任务和验收标准写入 `docs/tasks.md` 和 `docs/acceptance.md`
2. Developer Agent 读取任务文档，完成任务后写入 `reports/dev/task-xxx-dev-report.md`
3. Tester Agent 读取任务文档和开发报告，完成验收后写入 `reports/test/task-xxx-test-report.md`
4. 主 Agent 读取测试报告，更新任务状态，记录到 `worklog.md`

**注意事项**：
- 文件路径必须规范统一
- 文件格式必须标准化（Markdown）
- 每个 Agent 只读取自己需要的文件
- 不要假设其他 Agent 记住了什么

**相关任务**：
- 任务 ID: T001-T012
- 形成时间: 2026-04-28 至 2026-04-29

### 1.2 每个环节都有明确的输入输出文件

**经验描述**：
为每个 Agent 角色定义明确的输入输出文件，避免信息传递遗漏。文档化的交接比口头传递更可靠。

**适用场景**：
- 定义 Agent 职责时
- 设计协作流程时
- 排查信息传递问题时

**实施步骤**：
1. 为每个 Agent 定义"必读文件"列表
2. 为每个 Agent 定义"输出文件"列表
3. 确保"必读文件"在上游 Agent 的"输出文件"中
4. 在 Agent 模板中明确列出这些文件

**示例**：
- **Developer Agent 必读**：`docs/tasks.md`、`docs/acceptance.md`、`memory/lessons.md`、`memory/pitfalls.md`
- **Developer Agent 输出**：`reports/dev/task-xxx-dev-report.md`、修改后的业务代码
- **Tester Agent 必读**：`docs/tasks.md`、`docs/acceptance.md`、`reports/dev/task-xxx-dev-report.md`、业务代码
- **Tester Agent 输出**：`reports/test/task-xxx-test-report.md`

## 2. 主 Agent 调度经验

### 2.1 主 Agent 只负责调度和决策，不直接编码、不直接测试

**经验描述**：
主 Agent 应该专注于流程控制和协调决策，不应该直接编码或测试。编码交给 Developer，测试交给 Tester，主 Agent 只做调度。

**适用场景**：
- 设计主 Agent 职责时
- 避免主 Agent 过载时
- 确保 Agent 职责清晰时

**实施步骤**：
1. 主 Agent 从 `docs/tasks.md` 选择下一个任务
2. 主 Agent 将任务分配给 Developer Agent
3. 主 Agent 等待 Developer 返回开发报告
4. 主 Agent 将任务分配给 Tester Agent
5. 主 Agent 根据测试结果决定：更新状态完成任务 或 进入修正循环
6. 主 Agent 更新 `docs/tasks.md` 和 `worklog.md`

**注意事项**：
- 不要让主 Agent 直接修改业务代码
- 不要让主 Agent 直接执行测试
- 主 Agent 的价值在于流程控制，不在于代码实现

**相关任务**：
- 任务 ID: T001-T012
- 形成时间: 2026-04-28 至 2026-04-29

---

### 2.2 状态更新应由主 Agent 统一管理

**经验描述**：
任务状态的更新应该由主 Agent 统一管理，不应该由 Developer 或 Tester 修改。这样可以避免状态冲突，确保状态更新的一致性。

**适用场景**：
- 任何需要更新任务状态的多 Agent 协作项目
- 需要避免状态冲突时

**实施步骤**：
1. 只在主 Agent 的职责中包含"更新任务状态"
2. 在 Developer 和 Tester 的规范中明确禁止修改 `docs/tasks.md`
3. 主 Agent 只在测试通过后才更新状态为 `completed`
4. 状态更新后立即记录到 `worklog.md`

**注意事项**：
- Developer 完成任务后，状态仍然是 `pending` 或 `in_progress`，只有 Tester 通过后才改为 `completed`
- 不要让 Developer 自己判断任务完成
- 不要让 Tester 修改任务状态

## 3. Developer Agent 开发经验

### 3.1 Developer 必须遵守最小必要修改

**经验描述**：
Developer 只应该做当前任务需要的修改，不应该提前实现后续功能，不应该擅自添加"锦上添花"的功能。验收标准写什么，就做什么。

**适用场景**：
- 所有开发任务
- 特别是任务粒度较细、分步实施的项目

**实施步骤**：
1. 仔细阅读验收标准，列出所有需要做的内容
2. 只做验收标准中明确列出的内容
3. 验收标准没有提到的功能，坚决不做
4. 完成后自测，确保只做了要求的内容

**示例**：
- ✅ 正确：任务要求"添加输入框"，就只添加 `<input id="taskInput">`
- ❌ 错误：任务要求"添加输入框"，却同时添加了按钮和列表

**注意事项**：
- 不要认为"反正迟早要做"
- 不要认为"这样更好"
- 不要认为"顺手加上"

**相关任务**：
- 任务 ID: T001-T011
- 形成时间: 2026-04-28 至 2026-04-29

---

### 3.2 开始开发前必须读取 memory 文件

**经验描述**：
Developer 在开始任务前，必须读取 `memory/lessons.md` 和 `memory/pitfalls.md`，了解已有经验和已知陷阱，避免重复踩坑。

**适用场景**：
- 所有开发任务
- 特别是可能有历史遗留问题的任务

**实施步骤**：
1. 读取 `memory/lessons.md`，了解有效经验
2. 读取 `memory/pitfalls.md`，了解已知陷阱
3. 检查当前任务是否可能遇到类似问题
4. 应用已有经验，避免重复踩坑

**注意事项**：
- memory 文件应该在 Agent 模板的"必读文件"列表中
- 如果发现了新问题，建议更新 memory 文件

## 4. Tester Agent 验收经验

### 4.1 Tester 必须独立验收，不能盲信 Developer 报告

**经验描述**：
Tester 不能只看开发报告就相信实现了功能，必须读取实际代码并逐项对照验收标准。独立验收是多 Agent 协作质量保证的关键。

**适用场景**：
- 所有测试验收任务
- 特别是重构类任务

**实施步骤**：
1. 读取开发报告，了解 Developer 声称做了什么
2. 读取业务代码，确认实际做了什么
3. 逐项对照验收标准，每项都要测试
4. 检查历史功能是否被破坏（重构任务）
5. 判定 PASS 或 FAIL

**核心规则**：
不能只看开发报告就 PASS；必须读取实际代码，并逐项对照验收标准。

**相关任务**：
- 任务 ID: T012
- 形成时间: 2026-04-29

---

### 4.2 Tester 发现问题必须提供详细反馈

**经验描述**：
当 Tester 判定 FAIL 时，必须在测试报告中提供详细反馈：问题描述、复现方式、影响范围、建议修复方向。详细的反馈可以帮助 Developer 快速定位和修复问题。

**适用场景**：
- 所有 FAIL 的测试报告
- 特别是复杂 bug 的反馈

**实施步骤**：
1. 问题描述：哪项验收标准未通过，具体问题是什么
2. 复现方式：详细的操作步骤，如何复现问题
3. 影响范围：问题影响了什么功能，是否影响其他功能
4. 建议修复方向：可以提供具体的修复建议

**示例**：
```markdown
### 问题 1: 空输入校验不生效

**问题描述**:
验收标准要求"空输入不能添加任务"，但实际空输入时仍然可以添加。

**复现方式**:
1. 打开页面
2. 不输入任何内容
3. 点击"添加"按钮
4. 观察到空任务被添加到列表

**影响范围**:
- 影响数据质量
- 可能导致用户体验问题

**建议修复方向**:
建议在 `addTask()` 函数中添加空输入校验
```

**相关任务**：
- 任务 ID: T012
- 形成时间: 2026-04-29

## 5. 修正循环经验

### 5.1 最多 3 轮修正，避免无限循环

**经验描述**：
修正循环应该限制最大轮次，当前规则是最多 3 轮。第 3 轮失败后，主 Agent 介入处理，避免陷入无限循环。

**适用场景**：
- 所有修正循环
- 特别是复杂 bug 的修复

**实施步骤**：
1. 第 1 轮修复后仍失败 → 第 2 轮
2. 第 2 轮修复后仍失败 → 第 3 轮
3. 第 3 轮修复后仍失败 → 主 Agent 介入
4. 主 Agent 评估问题，决定是否需要调整验收标准或需求

**注意事项**：
- 每轮修复都要在开发报告中追加"修复记录"
- 每轮修复后都要由原 Tester 复测
- 如果 3 轮后仍失败，可能需要重新审视需求或验收标准

**相关任务**：
- 任务 ID: T012
- 形成时间: 2026-04-29

---

### 5.2 谁写的 bug 谁修复，谁提出的 bug 谁复测

**经验描述**：
修正循环时，应该回到原 Developer Agent 修复，由原 Tester Agent 复测。这样可以确保上下文连续性，提高修复效率。

**适用场景**：
- 所有修正循环
- 需要 resume Agent 上下文的场景

**实施步骤**：
1. 主 Agent 通过 DEV_ID resume 原 Developer Agent
2. 原 Developer 阅读测试报告，修复问题
3. 原 Developer 更新开发报告，追加修复记录
4. 主 Agent 通过 TEST_ID resume 原 Tester Agent
5. 原 Tester 阅读修复记录，复测问题
6. 重复直到 PASS 或达到 3 轮上限

**注意事项**：
- 需要在开发报告和测试报告中记录 Agent ID（DEV_ID、TEST_ID）
- 如果无法 resume 原 Agent，需要向主 Agent 报告

**相关任务**：
- 任务 ID: T012
- 形成时间: 2026-04-29

## 6. 报告与日志经验

### 6.1 子 Agent 返回主 Agent 的内容必须简洁

**经验描述**：
子 Agent（Developer、Tester）返回给主 Agent 的内容必须简洁，只返回关键信息（报告路径、状态、Agent ID）。详细内容应该写入报告文件，不要直接返回给主 Agent。

**适用场景**：
- 所有子 Agent 向主 Agent 汇报时
- 避免主 Agent 上下文过载时

**实施步骤**：
1. **Developer Agent 输出格式**：
   ```
   DEV_REPORT=reports/dev/task-xxx-dev-report.md
   STATUS=done
   DEV_ID=[实际的 Agent Session ID 或 UNKNOWN]
   ```

2. **Tester Agent 输出格式**：
   ```
   TEST_REPORT=reports/test/task-xxx-test-report.md
   RESULT=pass
   TEST_ID=[实际的 Agent Session ID 或 UNKNOWN]
   ```

3. 将所有详细信息写入对应的报告文件

**注意事项**：
- 不要返回大段代码
- 不要返回大段解释
- 详细内容写入报告，简洁格式返回给主 Agent

**相关任务**：
- 任务 ID: T001-T012
- 形成时间: 2026-04-28 至 2026-04-29

### 6.2 worklog.md 用于记录整体协作过程

**经验描述**：
`worklog.md` 用于记录整体协作过程，包括每个任务的执行过程、交付成果、测试结果、技术亮点等。这是项目级的工作日志，不是单任务的报告。

**适用场景**：
- 所有需要记录整体协作过程的项目
- 需要复盘协作流程时

**实施步骤**：
1. 每个任务完成后，由主 Agent 更新 `worklog.md`
2. 记录任务 ID、任务名称、状态、优先级、负责人
3. 记录执行过程、交付成果、测试结果
4. 记录验收标准检查项、前序任务保持情况
5. 可选：记录技术亮点

**注意事项**：
- `worklog.md` 是项目级的，不是单任务的
- 开发报告和测试报告是单任务的，放在 `reports/dev/` 和 `reports/test/`

## 7. 重构任务经验

### 7.1 重构任务必须做历史功能回归检查

**经验描述**：
重构类任务必须做完整的回归测试，确保所有历史功能没有被破坏。Task 012 证明，重构很容易无意中改变历史功能行为。

**适用场景**：
- 所有重构类任务（代码重构、样式重构）
- 修改核心逻辑的任务
- 修改公共函数的任务

**实施步骤**：
1. 重构前，列出所有相关历史功能
2. 重构时，只改代码结构，不改功能行为
3. 重构后，逐项测试所有历史功能
4. 特别关注边界情况和用户输入保留情况

**回归测试清单**：
- [ ] 测试所有已实现的功能
- [ ] 测试边界情况
- [ ] 测试用户输入的保留情况
- [ ] 测试数据结构是否改变
- [ ] 测试 API 调用是否改变

**相关任务**：
- 任务 ID: T012
- 形成时间: 2026-04-29

### 7.2 Task 012 证明 Tester 独立验收有价值

**经验描述**：
Task 012 中，Developer 在重构时无意中改变了历史功能行为（将 `taskInput.value` 改为 `taskInput.value.trim()`），Tester 通过独立验收发现了这个问题。这证明了独立验收的价值。

**适用场景**：
- 重构类任务
- 需要保证历史功能不被破坏时
- 评估测试流程价值时

**经验要点**：
- Developer 完成不等于任务完成
- Tester 独立验收可以发现 Developer 自己忽略的问题
- 重构任务特别需要 Tester 的独立检查
- 行为一致性检查比功能检查更重要

## 8. Skill 候选沉淀经验

### 8.1 只记录与多 Agent 协作相关的 Skill 候选

**经验描述**：
Skill 候选库应该只记录服务于多 Agent 协作的流程、检查清单、报告模板，不应该记录普通的代码技巧。避免 Skill 候选库变成代码技巧收集库。

**适用场景**：
- 记录 Skill 候选时
- 评估哪些内容值得记录时

**实施步骤**：
1. 判断候选内容是否服务于多 Agent 协作
2. 判断候选内容是否具有复用价值
3. 判断候选内容是否可以抽象为标准化流程
4. 如果与协作无关，即使是有用的代码技巧，也不记录

**应该记录**：
- 标准化的流程（如：任务分配流程、验收流程）
- 检查清单（如：开发检查清单、测试检查清单）
- 报告模板（如：开发报告模板、测试报告模板）
- 修正循环方法

**不应该记录**：
- 普通的代码技巧或优化方法
- 与多 Agent 协作无关的技术细节
- 可以在通用编程资料中找到的知识

**相关任务**：
- 任务 ID: T016
- 形成时间: 2026-04-29

### 8.2 复杂规则应先写入文档，再固化到 Agent 模板

**经验描述**：
当发现复杂的协作规则时，应该先写入 `docs/workflow.md` 这样的文档，经过验证后再固化到 Agent 模板（`.claude/agents/developer.md`、`.claude/agents/tester.md`）。文档先行，模板固化。

**适用场景**：
- 发现新的协作规则时
- 优化现有流程时
- 创建新的 Agent 模板时

**实施步骤**：
1. 先在 `docs/workflow.md` 中记录规则
2. 在实际项目中验证规则的有效性
3. 根据验证结果调整规则
4. 将验证后的规则固化到 Agent 模板
5. 确保 Agent 模板与文档保持一致

**注意事项**：
- 文档是规则的表达，模板是规则的固化
- 文档和模板应该保持同步
- 修改规则时，先改文档，再改模板

## 9. 可复用原则总结

### 9.1 核心 Agent 职责划分原则

**经验描述**：
多 Agent 协作的核心原则是职责清晰、边界明确。每个 Agent 只做自己职责范围内的事情，不越界、不替代。

**核心原则**：
- **主 Agent**：调度和决策，不编码、不测试
- **Planner Agent**：任务拆分和验收标准制定，不编码
- **Developer Agent**：开发一个明确任务，不测试、不修改状态
- **Tester Agent**：独立验收，不修改代码、不修改状态

**适用场景**：
- 设计多 Agent 协作流程时
- 调整 Agent 职责时
- 排查职责混乱问题时

**相关任务**：
- 任务 ID: T001-T012
- 形成时间: 2026-04-28 至 2026-04-29

### 9.2 Developer 完成不等于任务完成

**经验描述**：
Developer 完成开发只是任务完成的第一步，必须经过 Tester 独立验收并 PASS 后，任务才算真正完成。这个原则保证了多 Agent 协作的质量。

**适用场景**：
- 所有开发任务
- 评估任务完成状态时

**核心要点**：
- Developer 完成 → 生成开发报告 → 等待 Tester 验收
- Tester 验收 → 生成测试报告 → 判定 PASS 或 FAIL
- 只有 Tester PASS 后，任务才算真正完成
- 如果 Tester FAIL，应进入修正循环

**相关任务**：
- 任务 ID: T001-T012
- 形成时间: 2026-04-28 至 2026-04-29

### 9.3 三个 memory 文件的分工

**经验描述**：
三个 memory 文件有明确的分工，各司其职，避免内容混杂。

**文件分工**：
- **`memory/lessons.md`**：记录已经验证有效、后续值得复用的经验（本文档）
- **`memory/pitfalls.md`**：记录踩坑和风险，包括错误案例和避免方法
- **`memory/skills-candidates.md`**：记录未来可能抽象成 Skill 的候选

**适用场景**：
- 记录经验时
- 评估应该记录到哪个文件时
- 避免内容混杂时

**相关任务**：
- 任务 ID: T013-T016
- 形成时间: 2026-04-29

### 9.4 Agent ID 记录有助于修正循环追踪责任

**经验描述**：
在开发报告和测试报告中记录 Agent ID（DEV_ID、TEST_ID），有助于在修正循环时 resume 同一个 Agent，保持上下文连续性。

**适用场景**：
- 所有需要生成报告的任务
- 可能进入修正循环的任务
- 需要追踪责任时

**实施步骤**：
1. 在开发报告的"任务信息"章节中记录 DEV_ID
2. 在测试报告的"测试信息"章节中记录 TEST_ID
3. 如果当前环境无法获取真实 Agent ID，先记录 UNKNOWN
4. 修正循环时，通过 ID resume 同一个 Agent

**相关任务**：
- 任务 ID: T012
- 形成时间: 2026-04-29
